A stopped container still exists.
docker ps shows only running ones. Use docker ps -a to see the stopped — they keep their name and disk until you rm.
It lives only as long as its main process.
A container isn't a box that stays open. When its main process exits, it stops. That's why docker run ubuntu seems to "do nothing" — give it -it … bash.
Model 06 · The kernel features behind the magic
06 / 07
There is no magic — just three kernel features
A container is an ordinary Linux process wearing three pieces of equipment.
ISOLATION
Namespaces
“what a process can see”
Partition kernel resources so a container sees its own slice — and nothing of the host.
PIDown process tree (your app is PID 1)
NETown interfaces, IP, routing
MNTown filesystem mounts (own /)
UTSown hostname
IPCown inter-process comms
USERmap container users → host users
LIMITS
Control Groups
“how much it can use”
Meter and cap resource usage — e.g. “at most 512 MB RAM and half a CPU core.”
CPU0.5 core
Memory512 MB cap
Disk I/Othrottled
Networklimited
STORAGE
Union FS
“how images are layered”
OverlayFS merges stacked read-only layers into one view, adding a writable layer on run.
writable layer (per container)
RO layer · app
RO layer · libs
RO layer · base
Common layers are shared across images — stored once, never duplicated.
Mental model:
Namespaces = what you see　·　cgroups = what you can use　·　Union FS = how the image is assembled
Model 07 · The writable layer is disposable
07 / 07
Statelessness — each container starts fresh
Changes live in the writable layer. Remove the container and they're gone.
Container A
from ubuntu
apt-get install curl ✓
writable layer · has curl ✓
shared read-only image
Container B
same image, new run
curl: not found ✗
writable layer · empty
shared read-only image
docker rm A
cleanup
writable layer destroyed
curl gone with it ✗
image untouched, reusable
Whatever you change inside a container lives only in that one container's writable layer. A new container from the same image is born clean — and removing a container deletes its layer for good.

The seven mental models
recap
Carry these seven sentences forward
01
A container is an isolated process sharing the host kernel — not a lightweight VM.
02
An image is the read-only template (class / recipe); a container is a running instance (object / dish).
03
Four parts: client talks over REST to the daemon, which uses the registry and manages objects.
04
docker run = pull (if needed) + create + start, all in one line.
05
A container lives only as long as its main process; stopped ≠ removed (ps -a).
06
Namespaces = see, cgroups = use, Union FS = stack.
07
Containers are stateless — changes live in the disposable writable layer; persistence needs volumes (Day 3).
